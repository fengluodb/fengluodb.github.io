    本文介绍6.824的第一个实验 —— MapReduce的实现过程。

    MapReduce概述

    MapReduce是一种编程模型，是处理和生成大数据集合的相关实现。用户指定一个map函数，它将一个K/V对处理生成一个中间结果的K/V对集合；以及一个reduce函数，它将中间结果中所有key相同的K/V对的value进行合并。

    编程模型

    MapReduce的计算过程就是由输入的K/V对集合生成输出K/V对集合。MapReduce库的用户把计算过程表达为两个函数：Map和Reduce。

    Map：由用户编写，根据输入的一个K/V对生成一组中间结果K/V对。MapReduce库负责将key相同的中间结果的value汇集在一起并把它们传递给Reduce函数。

    Reduce：也是由用户编写，接收中间结果的key和以这个key为键的所有中间结果的value的集合。Reduce函数合并这些value，产生一个更小的value集合。最典型的情况是每次调用Reduce输出0个或1个值。中间结果通过迭代器的方式被用户的Reduce函数访问，避免了当中间结果的value集合比较大时无法将它们全部装入内存。

    示例：考虑一个问题，给定一个大规模的文档集合，统计每个单词的出现次数。用户会写出类似下面的伪代码：

    map(String key, String value):
  // key: document name
  // value: document contents
  for each word w in value:
    EmitIntermediate(w, "1");

reduce(String key, Iterator values):
  // key: a word
  // values: a list of counts
  int result = 0;
  for each v in values:
    result += ParseInt(v);
  Emit(AsString(result));

    分步和可靠性

    MapReduce通过把对数据集的大规模操作分发给网络上的每个节点实现可靠性；每个节点会周期性的把完成的工作和状态的更新报告回来。如果一个节点保持沉默超过一个预设的时间间隔，主节点记录下这个节点状态为死亡，并把分配给这个节点的数据发到别的节点。

    注：该实验中无需考虑主节点死亡的情况。

    实验要求

    Your job is to implement a distributed MapReduce, consisting of two programs, the coordinator and the worker. There will be just one coordinator process, and one or more worker processes executing in parallel. In a real system the workers would run on a bunch of different machines, but for this lab you’ll run them all on a single machine. The workers will talk to the coordinator via RPC. Each worker process will ask the coordinator for a task, read the task’s input from one or more files, execute the task, and write the task’s output to one or more files. The coordinator should notice if a worker hasn’t completed its task in a reasonable amount of time (for this lab, use ten seconds), and give the same task to a different worker.

    你的工作是实现一个分布式的MapReduce。它由两个程序组成，coordinator（协调者）和 worker（工作者）。实验中只有一个协调者进程，但有一个或多个并行执行的工作者进程。在一个真实的系统中，工作者进程会在一堆不同的机器上运行，但在这个实验中，你将在一台机器上运行它们。工作者将通过RPC与协调者对话。每个工作进程将向调度者请求一个任务，从一个或多个文件中读取任务的输入，执行任务，并将任务的输出写入一个或多个文件中。如果一个工作进程在合理的时间内没有完成其任务（在本实验中，使用10秒），协调者应该注意到，并将相同的任务交给不同的工作进程。

    We have given you a little code to start you off. The “main” routines for the coordinator and worker are in main/mrcoordinator.go and main/mrworker.go; don’t change these files. You should put your implementation in mr/coordinator.go, mr/worker.go, and mr/rpc.go.

    我们已经给了你一点代码，让你开始工作。协调者和工作者的 “主要 “例程在main/mrcoordinator.go和main/mrworker.go中，不要改变这些文件。你应该把你的实现放在mr/coordinator.go、mr/worker.go和mr/rpc.go。

    代码思路

    通过我对MapReduce的简要介绍和实验要求，我们可以理清每个文件中应实现的功能。

    首先，coordinator.go对应MapReduce中的主节点，它需要实现以下功能：

    1. 当worker请求任务时，将数据分发给它；
    2. 记录每个worker的工作时间，如果worker长时间未响应，默认worker死亡，重新分配任务。

    worker.go对应MapReduce中的工作者节点，它需要实现以下功能：

    1. 定期向主节点请求任务；
    2. 当任务完成后，向主节点发送完成的工作和状态更新。

    rpc.go是lab1的要求，worker通过rpc与coordinator对话，所以我们只需在rpc.go定义自己的RPC参数即可。

    代码实现

    rpc.go实现

    定义了我们的RPC后，我们才能知道worker.go和coordinator.go中需要如何请求和回复。所以，我们先从rpc.go开始实现。

    请读者自行思考一分钟，worker该如何发送什么数据，coordinator需要回复什么数据。

    首先，我们需要给每个任务分配一个Id，这样Coordinator才能知道自己分配了哪些任务，worker才能回复自己完成了什么任务。

    给任务分配了Id后，我们还需要给不同的任务分配不同的参数。一个Map任务，coordinator只需要提供一个源文件的地址即可（Worker完成map任务后，除了任务Id外，还需回传中间文件的地址）；而一个Reduce任务，我们需要提供任务对应的全部中间文件的地址。

    type Args struct {
	Index int // Index为-1代表请求任务，大于0时代表对应任务完成

	// 当worker完成map时传回的参数
	IntermediateFiles map[int]string
}

type Reply struct {
	Index int // 任务对应的编号

	// 当前任务为map时传给worker的参数
	File    string // 文件名
	NReduce int

	// 当前任务为reduce时传给worker的参数
	IntermediateFiles []string // 存放中间键的文件
}

    注：按照实验要求，map阶段应该把中间键分成nReduce个中间文件，其中nReduce是reduce任务的数量。所以Reply中增加了NRdcue参数。

    The map phase should divide the intermediate keys into buckets for nReduce reduce tasks, where nReduce is the number of reduce tasks – argument that main/mrcoordinator.go passes to MakeCoordinator(). So, each mapper needs to create nReduce intermediate files for consumption by the reduce tasks.

    coordinator实现

    定义好RPC后，我们开始在coordinator.go中实现对应的函数。我将这个函数命名为Task。

	if args.Index == -1 { // worker请求任务
		// 你的代码
	} else {  // worker回复该任务已完成
        // 你的代码
    }
	return nil

    注：请求任务时，需要根据目前状态提供参数，如果是MapReduce还在Map阶段，则提供File和NReduce；如果在Reduce阶段，则提供对应任务所需的中间文件的位置——IntermediateFiles。回复任务完成时，你需要标注对应任务已完成，如果当前状态在Map阶段，还需要处理回复的IntermediateFiles 。

    实现了Task后，我们只是完成了分配任务的功能。并没有实现容错功能——如果worker长时间未响应，默认worker死亡，重新分配任务。所以，我们还需在后台运行一个goroutine，定期检查哪些任务长时间未完成，并将它们重新分配。

    我将这个函数命名为Check

    func (c *Coordinator) Check() {
    for {
        time.Sleep(500 * time.Millisecond)  // 每隔0.5s检查是否有任务超时未完成
        // 你的代码
    }
}

    因此，我们的Coordinator结构体可定义为：

    type Coordinator struct {
	// Your definitions here.
	mu                sync.Mutex
	flag              int // 0代表map阶段，1代表reduce阶段，2代表所有工作完成
	files             []string

    Task是我自定义的结构体：

    type Task struct {
	index  int  // 任务Id
	status int  // 任务的状态
    time   time.Time // 任务的起始时间
}

    Coordinator定义完成，我们向Task中加入具体代码：

    当给worker分配任务时：

	task := c.wait[0] // 从wait中取出任务
    c.wait = c.wait[1:]

	task.time = time.Now()

    当收到worker回复任务完成的请求时：

    首先，判断该任务是否已经完成过，如果没执行过在done中添加，并在doing中标识该任务已完成。

    isvalid := true
for i := 0; i < len(c.done); i++ {
	if c.done[i].index == args.Index {
		isvalid = false
		break
	}
}

if isvalid {
	// 如果该任务之前没有执行过，在done中添加
	task := Task{
		index:  args.Index,
		status: Completed,
	}
	c.done = append(c.done, task)
    
    // 在doing中标识该任务已完成，待Check处理
    for i := 0; i < len(c.doing); i++ {
		if c.doing[i].index == args.Index {
			c.doing[i].status = Completed
			break
		}
	}
}

    如果当前状态是map，我们还需更新intermediateFiles参数。

    if args.IntermediateFiles != nil && c.flag == 0 { // worker完成map后传回的参数
	// 判断该map任务是否已经执行过，如果未执行过，更新intermediateFiles
	if isvalid {
		for k, v := range args.IntermediateFiles {
			c.intermediateFiles[k] = append(c.intermediateFiles[k], v)
			// fmt.Printf("intermediateFiles[%d] length: %d\n", k, len(c.intermediateFiles[k]))
		}
	}
}

    这样我们的Task已经全部完成，接下来填充Check函数。

    首先，检查doing中是否有超时未完成的任务，如果有，将其重新分配给wait

    length := len(c.doing)
for i := 0; i < length; i++ {
	if c.doing[i].status == Completed {
		continue
	} else if c.doing[i].status == Doing && time.Now().Sub(c.doing[i].time) > 10*time.Second {
		c.doing[i].status = Wait
		c.wait = append(c.wait, c.doing[i])
	} else {
		c.doing = append(c.doing, c.doing[i])
	}
}
c.doing = c.doing[length]

    其次，因为Check在后台定期执行，我们还可将更新MapReduce状态的任务交给它；

    当wait与doing中没有任务时，更新当前状态

    if c.flag == 0 && len(c.wait) == 0 && len(c.doing) == 0 {
	c.flag = 1
	// 分配reduce任务
	c.done = []Task{}
	for k := range c.intermediateFiles {
		task := Task{
			index:  k,
			status: Wait,
			time:   time.Now(),

	}

	// fmt.Println("map ending")
} else if c.flag == 1 && len(c.wait) == 0 && len(c.doing) == 0 {
	c.flag = 2

    mrsequential.go源码解读

    src/main/mrsequential.go，它是mapreduce的简单单线程实现。通过对它的阅读，我们可以了解实验中如何完成Map和Reduce函数。

    mrsequential.go的代码十分简单，MapReduce的实现呐都在main函数中。

    下面的这段代码是单线程的Map实现。我们只需读取一个文件的全部内容——content，然后再对其调用Map函数——mapf进行处理即可，Map生成的新键值对保存在intermediate中。

    //
// read each input file,
// pass it to Map,
// accumulate the intermediate Map output.
//
intermediate := []mr.KeyValue{}
for _, filename := range os.Args[2:] {
	file, err := os.Open(filename)
	if err != nil {
		log.Fatalf("cannot open %v", filename)
	}
	content, err := ioutil.ReadAll(file)
	if err != nil {
		log.Fatalf("cannot read %v", filename)
	}
	file.Close()
	kva := mapf(filename, string(content))
	intermediate = append(intermediate, kva...)
}

    下面的这段代码是单线程的Reduce实现。首先，对得到的新键值对进行排序，然后将新键值对划分成多个集合，每个集合中的键值对必须具有相同的key，最后，再对每个集合调用Reduce函数——reducef进行合并处理，并将结果写入输出文件中。

    sort.Sort(ByKey(intermediate))

oname := "mr-out-0"
ofile, _ := os.Create(oname)

//
// call Reduce on each distinct key in intermediate[],
// and print the result to mr-out-0.
//
i := 0
for i < len(intermediate) {
	j := i + 1
	for j < len(intermediate) && intermediate[j].Key == intermediate[i].Key {
		j++
	}
	values := []string{}
	for k := i; k < j; k++ {
		values = append(values, intermediate[k].Value)  
	}
	output := reducef(intermediate[i].Key, values)

	// this is the correct format for each line of Reduce output.
	fmt.Fprintf(ofile, "%v %v\n", intermediate[i].Key, output)

	i = j
}

ofile.Close()

    了解了Map和Reduce如何实现，我们开始编写我们的DoM和DoR函数。

    DoM函数 和 DoR函数

    func DoM(mapf func(string, string) []KeyValue, filename string, NReduce, index int) {
	// 读取task输入
	file, err := os.Open(filename)
	if err != nil {
		log.Fatalf("cannot open %v", filename)
	}
	content, err := ioutil.ReadAll(file)
	file.Close()

	// 调用mapf函数对输入进行处理
	kva := mapf(filename, string(content))

	intermediates := make(map[int][]KeyValue)

	for _, v := range kva {
		// 根据Reduce worker的个数，将中间键进行分配
		intermediates[ihash(v.Key)%NReduce] = append(intermediates[ihash(v.Key)%NReduce], v)
	}

	files := make(map[int]string)

	// 将中间键写入对应的中间文件中
	for k, v := range intermediates {
		filename := fmt.Sprintf("mr-%d-%d", index, k)
		files[k] = filename
		file, err := os.Create(filename)
		if err != nil {
			log.Fatalf("cannot create %v", filename)
		}
		enc := json.NewEncoder(file)
		for _, kv := range v {
			enc.Encode(&kv)
		}
	}

	// 向coordinator发送RPC请求，代表对应任务已经完成
	args := Args{}
	reply := Reply{}
	args.Index = index  // index代表对应任务id
	args.IntermediateFiles = files  // files代表生成的中间文件位置
	call("Coordinator.Task", &args, &reply)
}

    上面的代码就是我的DoM函数，也就是处理Map任务的逻辑。可以看出前半部分代码与mrsequential.go中的map部分几乎一模一样，整体上只是多了将中间键按reduce worker数量进行分配后写入中间文件和任务完成后发送请求的操作。增加这些操作的根据是lab1的 A few rules 和 Hints 。

    • The map phase should divide the intermediate keys into buckets for nReduce reduce tasks, where nReduce is the number of reduce tasks – argument that main/mrcoordinator.go passes to MakeCoordinator(). So, each mapper needs to create nReduce intermediate files for consumption by the reduce tasks.

    map阶段应该把中间键分成nReduce个桶，其中nReduce是reduce任务的数量 – main/mrcoordinator.go传递给MakeCoordinator()的参数。因此，每个映射器需要创建nReduce中间文件，供reducd任务使用。

    • A reasonable naming convention for intermediate files is mr-X-Y, where X is the Map task number, and Y is the reduce task number.

    • The worker’s map task code will need a way to store intermediate key/value pairs in files in a way that can be correctly read back during reduce tasks. One possibility is to use Go’s encoding/json package. To write key/value pairs in JSON format to an open file:

        enc := json.NewEncoder(file)
  for _, kv := ... {
    err := enc.Encode(&kv)

    • The map part of your worker can use the ihash(key) function (in worker.go) to pick the reduce task for a given key.

    如果对此部分有所疑惑，可以去看看MapReduce论文的实现概览，你会发现这部分代码的功能在上面都有着对应，能加深你对代码和论文的理解。

    func DoR(reducef func(string, []string) string, intermediateFiles []string, index int) {
	// 从中间文件中读取中间键
	intermediate := []KeyValue{}
	for _, filename := range intermediateFiles {
		file, err := os.Open(filename)
		if err != nil {
			log.Fatalf("cannot open %v", filename)
		}
		dec := json.NewDecoder(file)
		for {
			var kv KeyValue
			if err := dec.Decode(&kv); err != nil {
				break
			}
			intermediate = append(intermediate, kv)
		}
	}

	oname := fmt.Sprintf("mr-out-%d", index)
	ofile, _ := os.Create(oname)

	sort.Sort(ByKey(intermediate))

	// 与mrsequential.go中的reduce部分类似
	// 聚集相同key的键值对，在对这些键值对进行reduce操作
	for i := 0; i < len(intermediate); {
		j := i + 1
		for j < len(intermediate) && intermediate[j].Key == intermediate[i].Key {
			j++
		}
		values := []string{}
		for k := i; k < j; k++ {
			values = append(values, intermediate[k].Value)
		}
		output := reducef(intermediate[i].Key, values)

		// this is the correct format for each line of Reduce output.
		fmt.Fprintf(ofile, "%v %v\n", intermediate[i].Key, output)

		i = j
	}
	ofile.Close()

    // 向coordinator发送RPC请求，代表对应任务已经完成
	args := Args{}
	reply := Reply{}
	args.Index = index  // index代表对应任务的id
	call("Coordinator.Task", &args, &reply)
}

    • The worker implementation should put the output of the X’th reduce task in the file mr-out-X.
    • A mr-out-X file should contain one line per Reduce function output. The line should be generated with the Go "%v %v" format, called with the key and value. Have a look in main/mrsequential.go for the line commented “this is the correct format”. The test script will fail if your implementation deviates too much from this format.
